item: H-CSM-5
date: 2026-04-16T12:00:00Z
summary: Implement the control service as a gRPC bridge over published sessions
context: The work item required a desktop-facing control service without introducing a second runtime backend. The existing launch runtime already published a registry, control socket, and observation socket for each session.
decision: Build the new gRPC service as an orchestration layer that proxies through the existing session registry, control socket methods, and observation stream. Allow `strait service start` to optionally own one managed launch session at startup while still exposing the same underlying session surfaces.
rationale: This keeps launch and proxy behavior in one runtime implementation, preserves the existing local contracts, and gives desktop clients a remote-capable gRPC API without forking session state into a separate backend.
