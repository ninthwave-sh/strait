# Refocus plan: container-scoped MITM control plane

## Decision

Drop the `nono`-centered host-scoped pivot.

Keep the learning, keep the useful runtime primitives from this branch, and re-anchor phase 1 around the architecture Strait was already growing into: container-scoped MITM plus a great control-plane UX.

The core call is now explicit:

- primary runtime story: `strait launch`
- primary trust story: session-local CA trusted inside the container only
- primary product wedge: live request-level network policy for a running agent session
- secondary mode: standalone `strait proxy` for debugging and integrations, not as the main product claim

## Why the pivot fails

The product promise is stronger than "run a local proxy":

1. see the real request at L7
2. apply policy to what the agent is actually doing
3. cover everything the agent and its subprocess tree does
4. avoid machine-wide CA trust

On the host, you cannot count on that combination. Some clients honor process-local trust overrides. Some do not. Some subprocesses inherit them cleanly. Some do not. That means a host-scoped MITM story becomes fragile exactly where the product needs to feel seamless.

The container changes that.

- we own the runtime boundary
- we can inject the session CA into the sandbox only
- we can force all egress through the proxy
- we can keep filesystem, process, and network policy in one model
- we do not need to ask users to install a machine-wide root CA

That is not a compromise. That is the architectural advantage.

## What we learned from `nono`

The `nono` investigation was still valuable.

What they do well:

- local sandbox ergonomics
- session discovery and detached operation
- credential isolation
- authenticated local proxy access
- profile and preset style UX
- clear docs about what is and is not enforced

What they do not solve for our wedge:

- generic host-scoped MITM across everything the agent might do
- a no-machine-wide-trust answer for arbitrary local subprocess trees

Why: `nono` mostly avoids the CA problem by avoiding generic MITM. Its networking model is CONNECT/domain filtering plus reverse-proxy credential routes, not decrypted request visibility for arbitrary HTTPS traffic.

So the right takeaway is:

- do not pivot the product boundary toward `nono`
- do steal the good operator ideas

Useful ideas to borrow:

- per-session local proxy authentication
- better preset and profile ergonomics for launching supported agent setups
- sharper session discovery and diagnostics
- clearer onboarding copy about trust and enforcement boundaries

## Product statement

Strait is a container-scoped policy platform for AI agents.

Phase 1 is not about beating debugging proxies at interception. It is about turning owned MITM visibility into a usable permission workflow for a running agent session.

The user moments to optimize are:

1. What just got blocked?
2. Why was it blocked?
3. What is the smallest rule that would unblock it?
4. Can I allow it once, for this session, or persist it without restarting the world?
5. Can I do that from outside the agent terminal?

If a feature does not improve one of those moments, it is secondary.

## Product boundaries

### Primary runtime

`strait launch` stays primary.

That is where the trust boundary is real:

- container network is isolated
- traffic is forced through the host proxy
- the session-local CA is trusted only inside the container
- the agent and its subprocess tree inherit the same boundary

### Secondary runtime

Standalone `strait proxy` remains useful infrastructure.

It should keep publishing session IDs, control sockets, and observation sockets because those are useful primitives for:

- debugging
- experiments
- future integrations
- daemon work

But it is not the main product claim, because host trust and subprocess coverage vary by client.

### Non-goals for this refocus

- machine-wide CA trust as a product requirement
- host-scoped generic MITM across arbitrary local tools
- replacing `nono` as a general runtime substrate
- removing containers from the public product story
- a `mitmproxy` adoption spike as the immediate next bet

## What this branch should say now

This branch should no longer say:

- container sandboxing is legacy
- standalone proxy is the primary runtime
- `nono` is the first-class consumer path
- phase 1 removes container and unified policy framing

It should say:

- the `nono` investigation clarified the boundary
- container-scoped trust is the reason Strait can make a stronger promise
- the useful proxy-session publication work stays, but as a shared primitive
- the roadmap is now container-first again, with control-plane UX as the main user-facing investment

## Refocused roadmap

1. Lock the container trust boundary and launch contract.
2. Emit blocked-request IDs, explanations, and minimal candidate exceptions.
3. Add live decision actions for running container-backed sessions.
4. Persist approved network exceptions as durable Cedar policy with live network reload.
5. Add a local control service that owns container-backed sessions for desktop clients.
6. Build the first desktop control plane shell on top of that local service.
7. Polish onboarding, launch presets, and a first dogfood flow, borrowing the best UX lessons from `nono` without changing the runtime boundary.

## Verification standard

The refocused roadmap is only successful if a supported agent flow can do all of this without machine-wide trust installation:

1. launch inside `strait launch`
2. trust the session CA inside the container only
3. make real HTTPS requests through the MITM proxy
4. block a request and explain why
5. allow once or session or persist from outside the agent terminal
6. repeat the request successfully under the chosen scope

That is the moat.

## Final decision rule

Continue only if the team is committed to this exact combination:

- container-scoped MITM
- no machine-wide trust requirement
- seamless coverage of the agent subprocess tree
- control-plane UX as the main user-facing differentiator

Do not drift back toward a host-scoped MITM story that cannot actually keep the trust boundary promise.
