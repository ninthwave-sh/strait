# Pivot plan: strait as a cross-platform live MITM proxy and policy control surface

## Recommendation

Yes, the right direction is an external UI plus a general MITM proxy, not another CLI-first sandbox workflow.

Recommendation on implementation approach: compose rather than rebuild.

- use an existing MITM engine unless a short spike proves it cannot support the required control-plane semantics
- current best candidate: `mitmproxy`
- cut credential injection from v1

The wedge is:

- `strait` owns a general man-in-the-middle proxy for request-level network policy
- `strait` owns a cross-platform desktop control plane for live network policy discovery and exception handling
- `nono` is one consumer / integration, not the product boundary
- `strait` focuses only on network policy
- everything else is removed, not deprioritized

If the team does not want to build that external control surface, the recommendation is to stop rather than keep building overlapping CLI/runtime features.

## Why this changed

The new `nono` evidence shows they are already ambitious across most of the workflow we originally thought was open:

- `nono learn` already covers path and network discovery
- `nono policy show|diff|validate` already covers policy introspection
- supervisor mode already covers runtime approvals and approval backends
- session lifecycle already covers running-session control and approval-boundary semantics
- networking already covers domain filtering, endpoint filtering for credential routes, upstream proxying, and proxy-backed auth
- registry/distribution and audit-sink ambitions are explicit in issues and docs

That means a second CLI-first operator flow is not a real wedge.

The remaining opening is the out-of-band experience plus request-level control:

- live visibility outside the agent terminal
- firewall-like prompts and notifications
- richer request-level exception UX
- live mutation of running-session network policy
- a precise answer to: "what exact exception unblocks this request?"

## What `nono` appears to be aiming for

### Strong ambition / clear ownership

- sandbox/runtime substrate
- filesystem and process policy
- policy/profile authoring and introspection
- learn/discovery loop
- session lifecycle and detached operation
- runtime approvals via supervisor backends
- domain-filtering proxy and reverse-proxy credential routes
- signed policy/profile/package distribution
- audit trail and external sinks

### Important limitation in current shape

`nono` networking today is still mostly CONNECT/domain-filtering and reverse-proxy credential routes. Their docs explicitly say the proxy does not see plaintext for generic CONNECT tunnels.

That leaves a narrower request-level policy opportunity, but also raises overlap risk because they are already thinking about:

- interactive permission mode: `#47`
- URL/parameter approval in the HTTP proxy: `#176`
- policy-based approval models: `#436`

So the safe conclusion is:

- `nono` is moving into the policy workflow area
- `strait` should not compete on general-purpose CLI/runtime policy tooling
- `strait` should only exist if it owns a distinct control surface and a distinct request-exception UX

## Existing solution landscape

### Existing tools already solve the raw interception problem

- `mitmproxy`
  - open source, mature MITM engine
  - HTTP/1, HTTP/2, HTTP/3, WebSockets
  - interception, replay, modification, filtering, scripting, web UI
- HTTP Toolkit
  - strong cross-platform desktop UX
  - breakpoints that pause live traffic for manual editing
  - rule-building and request/response modification
- Charles / Proxyman / Fiddler / Requestly
  - mature debugging proxies with polished traffic inspection and rewrite workflows
- Little Snitch
  - strongest reference for permission UX, alerts, temporary allows, profiles, and control-center mental model

### Where they still fall short for this exact problem

- they are mostly debugging tools, not agent-safe policy systems
- they do not give an opinionated observe -> ask -> enforce workflow for agent traffic
- they do not turn repeated traffic into minimal durable permission rules in an opinionated way
- they generally do not provide a permission-oriented control plane that answers:
  - why was this blocked?
  - what smallest rule would unblock it?
  - should this be once, session, or persistent?

Conclusion:

- the market is crowded for generic debugging proxies
- the opening is the control plane and policy workflow, not raw interception

## Overlap map

## Too overlapped to keep building

- container sandboxing
- filesystem policy
- process policy
- unified `http/fs/proc` product framing
- `strait launch` as the main runtime story
- generic session lifecycle features already covered by `nono`
- generic learn/generate/diff/show/validate as a standalone CLI product
- git-backed policy as a core selling point
- policy/package distribution ambitions

## Open enough to pursue

- out-of-band desktop UI for running sessions
- notifications and popup approvals outside the terminal
- live network exception handling while the agent is still running
- request-level policy decisions richer than simple domain allowlists
- block explanation plus minimal exception suggestion UX
- optional pause-and-decide behavior for blocked requests

## Borderline overlap, so keep secondary

- GitHub repo / PR workflow for persisted policy changes
- semantic policy review artifacts
- credential mediation

These can help the product, but they should not define phase 1.

## Ruthlessly simplified product statement

`strait` is not an agent sandbox.

`strait` is a cross-platform MITM proxy and desktop firewall for agent network requests.

It shows what a running agent needs, explains blocked requests, and lets the user allow the minimal network exception without living inside the agent terminal.

It should be usable by:

- `nono`
- other sandbox/runners
- direct local developer workflows
- eventually any tool that can be pointed at an HTTP(S) proxy

It should not try to beat existing tools at being a generic debugging proxy.

It should turn a generic MITM capability into a permission workflow for agents and automation.

## Core user problems to solve

1. "Why did the agent fail just now?"
2. "What exact network exception would unblock it?"
3. "Can I allow it while the session is still running?"
4. "Can I make this decision from outside the terminal?"

If a feature does not improve one of those four moments, cut it.

## Product shape

### Phase 1 product

A desktop companion app with:

- session list
- blocked-request feed
- per-request details
- allow once
- allow for session
- persist rule
- deny
- clear explanation of why it was blocked
- notification / popup entrypoint

The desktop app talks to a local `strait` daemon / control service.

The daemon owns:

- MITM proxying
- network event ingestion
- rule suggestion
- live network policy mutation
- optional request holding / resumption if technically feasible

### CLI role in this model

CLI is secondary.

Keep a CLI only as:

- local debugging surface
- automation surface
- fallback for non-desktop environments

Do not treat CLI as the product wedge.

The user is right that CLI-first only matters if it can control a running session from outside the agent terminal. That becomes a control-plane detail, not the main UX.

## Relationship to `nono`

`nono` should be treated as an early integration, not the center of the architecture.

Recommended framing:

- `strait` is a general network policy layer
- `nono` is one strong consumer because it already manages sandboxing and session lifecycle
- if `nono` grows more policy UX, `strait` still has a reason to exist as the external request-level control surface

This also reduces strategy risk:

- you are not betting the whole product on a single upstream
- you are not forced to mirror `nono`'s CLI or profile model
- you can support other agent runtimes later without changing the core thesis

## Electron vs native

The product call is correct: start with a cross-platform desktop shell.

Recommendation:

- commit to a desktop app / tray app first
- do not over-index on Electron specifically in the strategy
- choose the fastest stack that gives you:
  - tray/menu bar presence
  - native notifications
  - local IPC to a daemon
  - simple event streaming UI

Because this repo is Rust-heavy, a Rust-backed desktop shell such as Tauri may fit better operationally, but that is an implementation choice, not the product decision.

What matters strategically is:

- desktop control surface first
- cross-platform from the start if practical
- menu-bar / tray semantics on macOS and equivalent tray behavior elsewhere

## `mitmproxy` recommendation

Recommendation: build on top of `mitmproxy` unless a short spike disproves the fit.

Why:

- it already solves the hard non-differentiated proxy mechanics
- it has a strong extension model via addons and hooks
- it already supports the protocol surface this product needs
- the scarce engineering effort should go into policy semantics, eventing, and control-plane UX

When to reject `mitmproxy`:

- if its Python runtime / addon model becomes an operational liability for the intended distribution model
- if low-latency request hold / resume semantics are too awkward to implement cleanly
- if control over TLS / certificate lifecycle or embedding proves too constrained
- if the integration boundary becomes more complex than replacing the core

Plan assumption for now:

- `strait` owns the policy model, event model, and control surface
- `mitmproxy` owns MITM/TLS/protocol plumbing unless the spike shows otherwise

## Narrowest viable wedge

### v1

- network-only
- compose on top of an existing MITM core rather than building one from scratch
- running-session control
- block-and-explain
- allow once / allow for session / persist
- desktop notification when something blocks
- if pause-and-resume is hard, start with block-and-retry

### v1 non-goals

- filesystem policy
- process policy
- containers
- full sandbox ownership
- building a new MITM engine from scratch unless composition fails
- registry/distribution
- Git-backed policies as a primary workflow
- PR creation as phase-1 core
- broad GitHub management UX
- credential injection

## Technical direction

### Keep

- `src/mitm.rs`
  - keep only if it survives a compose-vs-rebuild decision after a `mitmproxy` spike
  - if `mitmproxy` is adopted, this likely becomes adaptation/integration logic rather than the long-term proxy core
- `src/config.rs`
  - only the parts needed for network policy loading and live mutation
- `src/observe.rs`
  - as the base event stream for desktop UI
- `src/replay.rs`, `src/explain.rs`, `src/diff.rs`
  - only if they directly help explain or persist network exceptions

### Delete or remove from product scope

- `src/launch.rs`
- `src/container.rs`
- filesystem/process policy framing in docs and UX
- templates and commands that mainly exist for container / fs / proc workflows
- any roadmap work that expands unified multi-domain policy

The simplification rule is strict:

- if code exists only to support sandbox/container/fs/proc product value, plan to delete it
- if code exists only to keep old positioning alive, delete it

Additional simplification rule:

- if code exists only because we assumed we had to own proxy mechanics, delete or replace it when composition is cheaper

## Critical files likely to change first

- `src/main.rs`
  - reshape CLI into a daemon/control-plane orientation instead of a container/orchestrator orientation
- `src/mitm.rs`
  - either narrow into a compatibility / integration layer or retire after `mitmproxy` spike
- `src/config.rs`
  - shrink to network-policy config and live mutation support
- `src/observe.rs`
  - make event model desktop-UI friendly
- `README.md`
  - rewrite around the desktop firewall/control-surface product

## Immediate research / spike before implementation

Run a short technical spike against `mitmproxy` to answer four questions:

1. Can it pause or hold matching requests cleanly long enough for an external decision?
2. Can it stream structured events to a local desktop control plane with low friction?
3. Can its addon model express once / session / persistent rule semantics cleanly?
4. Is packaging / certificate onboarding acceptable for the target user experience?

Only if the answer is no on critical path items should the team keep or extend a custom proxy core.

## Critical files likely to remove or retire

- `src/launch.rs`
- `src/container.rs`
- container-centric quick starts and architecture docs
- templates centered on filesystem/process/container access

## Execution plan

1. Rewrite product statement and non-goals around network-only live control.
2. Remove sandbox/container/fs/proc positioning from the roadmap.
3. Run a `mitmproxy` composition spike and decide compose vs rebuild.
4. Define a local daemon + desktop app architecture around the chosen MITM core.
5. Build blocked-request event streaming and live mutation for network policy.
6. Integrate one consumer first, with `nono` as the initial target.
7. Build desktop notifications and request decision UI.
8. Add persist-rule flow.
9. Revisit GitHub/PR workflow only after the live-control product proves useful.

## Verification

### Product verification

1. Start a consumer that routes through the `strait` proxy. Use `nono` as the first integration.
2. Trigger a blocked network request.
3. Verify the desktop app shows:
   - the blocked request
   - why it was blocked
   - the smallest proposed exception
4. Approve once and confirm the running session can continue.
5. Approve for session and confirm repeated matching requests work without restart.
6. Persist a rule and confirm the next session starts with the exception already in place.

### Technical verification

- `cargo test --all-features`
- `cargo clippy --all-features -- -D warnings`
- `cargo fmt --check`
- integration coverage for:
  - blocked network event emission
  - allow once / session allow / persist behavior
  - live policy mutation without restart
  - fallback block-and-retry flow
  - desktop-client to local-daemon IPC

## Risks

- `nono` may continue to move into approval UX, narrowing the wedge further.
- If `nono` adds a first-class external approval UI, the wedge may collapse.
- If request pause/resume is hard, the first version may need to rely on block-and-retry.
- If request-level decrypted visibility requires too much new backend work, the product could drift back into infrastructure instead of staying a focused control surface.
- If too much of the product is framed as `nono`-specific, the wedge will collapse into integration work rather than product differentiation.
- If `mitmproxy` cannot support the needed control semantics cleanly, the team may still face a custom-core decision after the spike.

## Final decision rule

Continue only if the team commits to this exact wedge:

- cross-platform desktop control surface
- general MITM proxy as the backend
- network-only
- live exception handling for running sessions
- ruthless deletion of sandbox/container/fs/proc scope
- composition first, rebuild only by exception

Do not continue if the plan drifts back toward:

- another CLI-first runtime
- another generalized policy platform
- partial retention of old sandbox/container scope

## Assumptions locked for this plan

- The differentiator is the combination of request-level MITM control plus out-of-band desktop UI, not the CLI.
- `nono` is one consumer, not the architectural center.
- Network policy is the only justified scope for phase 1.
- Git-backed policy and GitHub PR workflow are secondary, not core.
- `mitmproxy` is the default backend assumption until disproven by a spike.
- credential injection is out of scope for phase 1.
